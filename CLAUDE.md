# STWO Circuits — Circuit-Based STARK Prover & Verifier

STWO Circuits is StarkWare's circuit-based STARK prover and verifier built on top of the
[stwo](https://github.com/starkware-libs/stwo) core proof system. It provides a circuit DSL
for defining arithmetic constraints using gates, generates STARK proofs for Cairo CPU
execution traces, and includes a standalone STARK verifier. The
[stwo-cairo](https://github.com/starkware-libs/stwo-cairo) repo provides the Cairo-specific
prover/verifier that this repo depends on for Cairo AIR definitions and trace handling.
Soundness bugs are catastrophic and irreversible — an accepted invalid proof or
rejected valid proof is a protocol failure with no recovery path.

See the [Circle STARKs paper](https://eprint.iacr.org/2024/278) for the underlying proof system.

## Priority Contract

All agent behavior is governed by this hierarchy. No exception, no override.

1. **SOUNDNESS & SECURITY** — Cryptographic correctness is absolute.
2. **PRODUCTION QUALITY** — All tests pass. All gates hold. No guardrail bypassed.
3. **PERFORMANCE** — Prover performance is a correctness property. Regressions block.

## Mathematical Context

This project implements a circuit-based STARK proving system on top of the stwo core library.
The circuit DSL (`circuits` crate) allows defining arithmetic constraints using gates (Add,
Sub, Mul, etc.) which are compiled into an AIR (Algebraic Intermediate Representation). The
prover generates STARK proofs using FRI over the Mersenne31 field with CM31/QM31 extensions.
The verifier independently validates proofs using FRI decommitment, out-of-domain sampling,
and Merkle authentication paths.

The `cairo_air` crate integrates with the Cairo CPU AIR from stwo-cairo, enabling proving
and verification of Cairo program execution. This includes privacy-preserving circuit
verification features.

**Key references:**
- [Circle STARKs paper](https://eprint.iacr.org/2024/278)
- [Cairo paper](https://eprint.iacr.org/2021/1063)
- [STWO Whitepaper](https://github.com/starkware-libs/stwo/blob/dev/.agents/papers/llm/Stwo_Whitepaper.llm.md)

## Key Architectural Decisions

1. **Gate-based circuit DSL**: The `circuits` crate provides a builder pattern centered on
   `Context<Value>`, `Var`, and the `Gate` trait. A `Context` holds the circuit structure
   (list of gates) alongside optional concrete `QM31` values for each variable. When `Value`
   is `NoValue` you build just the circuit topology; when `Value` is `QM31` you also
   compute witness values as you go. The `eval!()` macro provides concise inline arithmetic:
   `eval!(ctx, ((a) * (b)) - (c))`.

2. **Gate types and the `Gate` trait**: Every gate implements `Gate` with three methods:
   `check()` (validates the constraint against concrete values), `uses()` (input variables),
   and `yields()` (output variables). The built-in gate types are:
   - **Add / Sub / Mul** — standard field arithmetic, each with two inputs and one output.
   - **PointwiseMul** — component-wise multiplication of the four M31 limbs inside a QM31,
     bypassing extension-field multiplication.
   - **Eq** — equality constraint between two variables. Yields nothing (both inputs are uses).
   - **Blake** — Blake2s hash gate: takes a vector of input words plus a byte-length and
     produces two output QM31 values.
   - **Permutation** — enforces that two variable lists are equal as multi-sets. Used when
     an operation reorders values (e.g., shuffling Blake state words) and the prover needs to
     prove the output is a permutation of the input without fixing the order.
   - **Output** — marks a variable as a public output of the circuit. Uses but does not yield.

3. **Yield / use invariant and `guess()`**: Soundness of the lookup argument requires that
   every variable appears **exactly once** as a yield across all gates. Arithmetic gates
   (Add, Sub, Mul, PointwiseMul) yield their output; Eq and Output yield nothing.
   `guess()` introduces an unconstrained prover-provided variable (used for division witnesses,
   external inputs, constants, etc.). Because guessed variables are not yielded by any gate,
   `finalize_guessed_vars()` adds a trivial `x + 0 = x` gate for each, ensuring the
   single-yield invariant holds. The `check_yields()` method verifies this invariant.
   `check_vars_used()` is a complementary sanity check ensuring no variable is silently
   dropped — every variable must be consumed by at least one gate or explicitly marked unused.

4. **Circuit components**: The `circuit_air` crate defines components for Eq, QM31Ops,
   Blake (Gate, Round, RoundSigma, G, Output), XOR verification (TripleXor32,
   VerifyBitwiseXor at 4/7/8/9/12 bits), and range checks (RangeCheck15, RangeCheck16) —
   each implementing constraint evaluation and interaction via the logup protocol.

5. **AIR code generation**: Component constraint evaluators and subroutines in `circuit_air`
   and `cairo_air` are generated by stwo-air-infra. Manual edits to generated files will be lost.

6. **Stable Rust compatibility**: All crates except `circuit-prover` and
   `circuits-stark-verifier-examples` must compile on stable Rust (1.88.0). This is enforced by CI.

7. **Optimized test profile**: `circuit-prover` and `stwo` are compiled at opt-level 3 even
   in the test profile to keep Blake-related tests performant.

8. **Circuit finalization and preprocessing**: The `circuit_common` crate bridges the circuit
   DSL and the prover. `finalize_context()` prepares a circuit for proving by hashing all
   constants (via Blake) and adding their hash as public outputs, then padding the Eq, QM31Ops,
   and Blake components to power-of-two sizes (minimum `N_LANES=16`). `add_zk_blinding()`
   injects random values into the QM31Ops and Eq traces to achieve zero-knowledge.
   `PreprocessedCircuit::preprocess_circuit()` then builds the fixed preprocessed trace. 
   The resulting `PreProcessedTrace` and `CircuitParams` are then passed to the
   prover unchanged.

## Key Metrics

The two primary metrics for this project are **prover time** (dominated by witness generation)
and **proof size**.

### Prover Time

Prover time is the wall-clock duration of `prove_circuit()` in `circuit_prover/src/prover.rs`.
The pipeline is:

1. **Witness generation** (`write_trace()` in `circuit_prover/src/witness/trace.rs`) —
   generates base traces for all circuit components. This is heavily parallelized using
   `rayon` and `std::thread::scope`, with SIMD vectorization (N_LANES=16 M31 values per
   register via `portable_simd`). ClaimGenerator initialization spawns parallel threads,
   and trace writing proceeds in multiple `rayon::join` stages.
2. **Interaction trace** (`write_interaction_trace()`) — computes logup interaction traces
   for all components.
3. **STARK proving** (`prove_ex()` from stwo core) — FRI commitment, Merkle tree
   construction, and query generation.

There is currently no built-in timing instrumentation in the codebase. To measure prover
time, wrap `prove_circuit()` calls with `std::time::Instant` or use an external profiler.
Because `circuit-prover` and `stwo` are compiled at opt-level 3 even in the test profile,
test-time measurements approximate release performance.

Performance regressions are treated as blocking issues (see Priority Contract).

### Proof Size

Proof size is measured in `u32` elements (multiply by 4 for bytes). The `ProofInfo` struct
in `stark_verifier/src/proof.rs` computes a full size breakdown from a `ProofConfig` without
needing an actual proof:

```rust
let info = ProofInfo::from_config(&config);
println!("{info}");          // human-readable breakdown
let total_bytes = info.total() * 4;
```

The proof size decomposes into:

| Component | Description | Scaling |
|-----------|-------------|---------|
| `fixed` | Channel salt, 4 roots, nonces | Constant (44 u32s) |
| `claim` | Packed enable bits, log sizes, claimed sums | O(n_components) |
| `oods` | Out-of-domain evaluations | O(n_columns) |
| `fri_commitments` | One hash per FRI layer | O(log(trace_size)) |
| `fri_last_layer` | QM31 coefficients | O(2^log_n_last_layer_coefs) |
| `eval_samples` | M31 samples at queried positions | O(n_columns * n_queries) |
| `eval_auth` | Merkle auth paths for eval domain | O(n_queries * log(eval_domain)) |
| `fri_auth` | Merkle auth paths for FRI layers | O(n_queries * sum(layer_depths)) |
| `fri_witness` | Coset values per FRI fold step | O(n_queries * sum(2^step)) |

The per-query components (`eval_samples`, `eval_auth`, `fri_auth`, `fri_witness`) dominate
for typical configurations. Reducing `n_queries` shrinks the proof but lowers security;
increasing `fold_step` reduces FRI auth path size at the cost of larger FRI witnesses.
